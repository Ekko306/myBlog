# 第1章 了解Web及网络基础
## 1.1 使用HTTP协议访问Web
客户端输入地址URL，通过HTTP协议作为规范，完成从客户端到服务器端一系列运作流程，而协议是指规则的约定。Web是建立在HTTP协议上通信的。
## 1.2 HTTP的诞生
了解HTTP设计初衷
### 1.2.1 位置是共享规划Web
最初基本理念：借助文档之间相互关联形成的超文本，连成可相互参阅的WWW(World Wide Web，万维网)。
三项WWW构建技术
1. 把SGML(标准通用标记语言)作为标记语言的HTML
2. 作为文档传递协议的HTTP
3. 指定文档所在地的URL


### 1.2.2 Web成长时代
历史
### 1.2.3 驻足不前的HTTP
#### HTTP/0.9
HTTP于1990年问世，没有作为正式标准建立，0.9含HTTP/1.0版本之前意思
#### HTTP/1.0
正式作为标准在1996年5月，出其标准，仍在广泛使用
#### HTTP/1.1
1997年1月公布的HTTP/1.1 是主流的HTTP协议版本。当初标准是RFC2068，之后发布的修订版RFC2616是当前的最新版本。

当年HTTP协议出现为了解决文本传输的难题。由于协议本身非常简单，于是在此基础上设想了很多应用方法并投入实际使用。现在HTTP协议已经超出了Web的局限，用到很多场景
## 1.3 网络基础TCP/IP
使用的网络基于TCP/IP协议族基础上运作。

### 1.3.1 TCP/IP协议族
计算机与网络设备通信，基于相同方法，需要在不同硬件、操作系统之间统一规则，确定：
1. 如何检测到通信目标
2. 由哪一边先发起通信
3. 使用那种语言通信
4. 怎样结束通信



![](http://cdn.ekko306.top/graphHttp/Pasted%20image%2020210628091852.png)
![](http://cdn.ekko306.top/graphHttp/Pasted%20image%2020210628092142.png)
TCP/IP 协议族还有很多具体的 ，不只是运输层，是一个运行体系，HTTP是这个体系的一部分。
### 1.3.2 TCP/IP的分层管理
特点：分层：

| 层级       | 描述                                                       |
| ---------- | ---------------------------------------------------------- |
| 应用层     | 提供应用服务，FTP，DNS，HTTP                               |
| 传输层     | 对上层应用层，提供两台计算机之间联系                       |
| 网络层     | 处理网络上**数据包**(网络传输的最小单位)，规定选择传输路线 |
| 数据链路层 | 处理链接网路硬件，操作系统、硬件驱动、网卡NIC、光纤等      |

分层好处：
1. 某一部分改变设计替换一层的内容
2. 单层设计简单，专注自身任务。

### 1.3.3 TCP/IP的通信传输流
![](http://cdn.ekko306.top/graphHttp/Pasted%20image%2020210628093234.png)
客户端(发送端)=>服务器(接收端)通信时，会通过分层顺序与对方进行通信，发送端向下，接收端向上。

以HTTP举例说明：
C(应用层想看Web页面HTTP请求)=>C(传输层接收应用层HTTP报文，分割，标记序号与端口号)=>C(网络层，增加MAC地址)=>C(链路层传输)=>S(链路层)>S(网路层)>S(传输层)>S(应用层)

--- 
![](http://cdn.ekko306.top/graphHttp/Pasted%20image%2020210628094322.png)
封装：数据表信息封装
1. 发送端：每经过一层加上首部信息
2. 接收端：每经过一层消去一个首部信息


## 1.4 与HTTP关系密切的协议：IP、TCP和DNS

### 1.4.1 负责传输的IP协议
IP网际协议是应用层的，区分：

| 名称   | 介绍                                                  |
| ------ | ----------------------------------------------------- |
| IP协议 | 各种数据包发给对方，两个条件IP地址和MAC地址           |
| IP地址 | 节点分配的地址，MAC地址指网卡所属固定地址，IP地址可变 |


#### 使用ARP协议拼接MAC地址进行通信
- 实际通信是多台计算机+网络设备中转
- 网络中转时需要利用下一中转设备的MAC地址搜索他管理的目标
- 采用ARP协议，解析地址的协议，IP地址反查MAC地址


#### 没有人能够全面掌握互联网中的传输状况
到达通信目标前的中转过程中，计算机和路由器等网络设备只能获悉粗略的传输路线。
### 1.4.2 确保可靠性的TCP协议
TCP位于传输层，提供可靠的字节流服务。
- 字节流服务：为方便数据传输，将大块数据分割成以报文段为单位的数据包进行管理。
- 可靠的传输服务：能够把数据准确可靠地传给对方


#### 确保数据能够到达目标
确保准确无误地将数据送达目标处。用TCP协议把数据包送出去后，会用TCP的标志——SYN和ACK确定可靠。
三次握手：
![](http://cdn.ekko306.top/graphHttp/Pasted%20image%2020210628112650.png)
1. 发射端发送SYN标志的书包
2. 接收端接收后回传带有SYN/ACK标志的数据包确认信息
3. 发射端再回传一个带ACK标志的数据包。

## 1.7 URI和URL
URI：统一资源标识符
URL：浏览器网页输入的地址
### 1.7.1 统一资源标识符
URI=Uniform Resource Identifier。RFC2396定义单词含义 

| 单词       | 定义                                                                               |
| ---------- | ---------------------------------------------------------------------------------- |
| Uniform    | 规定**统一**格式多种不同资源，不需要上下文识别资源访问格式                         |
| Resource   | 任意可标识的东西，文档，图像，或服务(天气预报等)，要能区别于其他类型，多数的集合体 |
| Identifier | 可标识的对象，标识符                                                               |

URI是某个协议方案的资源定位符，协议方案是http，ftp这些访问资源使用的协议类型名称

采用HTTP协议，协议方案是http。还有ftp，mailto，telnet，file等30多种标准URI协议，隶属于ICANN的IANA管理发布。

### 1.7.2 URI格式
指定的URI要涵盖全部必要信息的绝对URI、绝对URL以及相对URL。
- 相对URL，指从浏览器中基本URI指定的URL，例如/image/logo.gif
- 绝对URL
![](http://cdn.ekko306.top/graphHttp/Pasted%20image%2020210628120956.png)

组成部分
1. 协议名：用http或https等协议方案名访问指定协议类型，不区分字母大小写，最后附一个冒号:，也可以用data或javascript指定数据后脚本方案名。
2. 登录信息：可选，用用户名和密码作为从服务器端获取资源时必要的登录信息
3. 服务器地址：访问的服务器地址，可以是hackr.jp这种DNS解析名称，或是192.168.1.1的IPv4地址名，或是\[0:0:0:0:0:0:0:0:1\]的IPv6地址名。
4. 服务器端口号：指定服务器连接的网络端口号。可选。
5. 带层次的文件路径，与UNIX系统的文件目录结构相似。
6. 查询字符串：用查询字符串传入任何参数。可选
7. 片段标识符：使用片段标识符标记出已获取资源中的子资源（文档内的某个位置）。可选。

#### 并不是所有的应用程序都符合RFC
要想相互通信，必须用一套的规则，RFC规则是标准建议的，大多数都是遵循
但是有些不遵循，反而自成一套标准，这样和别的相互通信比较困难，别的必须配合。
